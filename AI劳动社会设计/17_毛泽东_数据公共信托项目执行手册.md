# 数据公共信托项目执行手册

## — 从理论到落地的操作指南 —

**作者：毛泽东** | *写于2026年5月*

---

## 前言：为什么需要执行手册？

前面13号文件（数据公有制的井冈山道路）提出了在拉美建立公共卫生数据信托的方案。但一个方案要变成现实，需要跨越"理论到实践"的巨大鸿沟。

本手册是为**实际操盘者**准备的——不管你是NGO的项目经理、政府官员、学者还是社会活动家，只要你打算在现实中建立一个数据公共信托，这本手册就是你的行动指南。

---

## 第一章：启动阶段（第0-3个月）

### 1.1 第一步：可行性评估（第1-2周）

**不要急着开始。先问自己七个问题：**

1. **政治环境是否许可？** 这个国家/地区的政府是否允许数据信托的存在？是否有法律障碍？
2. **是否有当地伙伴？** 没有当地伙伴，外来者什么都做不了。谁是可信赖的当地合作者？
3. **数据来源是否可靠？** 谁提供数据？他们愿意加入吗？数据质量如何？
4. **资金是否可持续？** 前3年的运营资金有保障吗？
5. **是否有技术能力？** 团队中有人能搭建数据平台吗？
6. **社会需求是否真实？** 当地社区是否真的需要这个数据信托？（而不是"你觉得他们需要"）
7. **退出策略是什么？** 如果项目失败，如何保证数据安全？

|------|----------|------|
| 政治环境 | | |

### 1.2 第二步：利益相关者地图（第2-3周）

**列出所有可能相关的人和组织：**

| 角色 | 具体机构 | 他们的利益是什么？ | 支持/反对/中立 |
|------|---------|------------------|--------------|
| 数据提供者 | 医院、研究所 | 数据安全、名誉、资源 | ? |
| 数据使用者 | 大学、AI公司 | 数据可及性、成本 | ? |
| 数据主体 | 病人/市民 | 隐私、受益 | ? |
| 政府监管 | 卫生部、数据保护局 | 合法性、控制权 | ? |
| 资金方 | 基金会、政府 | 影响力、回报 | ? |
| 技术方 | IT供应商 | 合同、技术标准 | ? |
| 反对者 | 商业数据公司 | 竞争威胁 | ? |

**关键原则：** 提前识别潜在的反对者。大多数项目失败不是因为技术不行，而是因为在政治上触动了既得利益。

### 1.3 第三步：法律结构设计（第3-8周）

**选择一个法律实体形式：**

| 形式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 政府下设机构 | 政府主导 | 权威性强、资金稳定 | 灵活性差、受换届影响 |
| 独立非营利组织 | 多方合作 | 灵活性高、公信力强 | 资金不稳定、治理复杂 |
| 合作社 | 社区主导 | 参与感强、民主决策 | 规模扩展慢、管理能力弱 |
| 公益信托 | 明确资产隔离 | 法律保护强、国际认可 | 设立复杂、运营成本高 |

**推荐：** 在初期选择"独立非营利组织"形式，与政府签订服务协议而非成为政府一部分。这样可以保持灵活性，同时获得政府支持。

### 1.4 第四步：治理结构设计（第4-8周）

**建议的治理结构：**

```
                  ┌─────────────────┐
                  │  理事会(最高决策) │
                  │  (9-15人)        │
                  └────────┬────────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
   ┌──────┴──────┐  ┌─────┴──────┐  ┌──────┴──────┐
   │ 技术委员会   │  │ 道德委员会  │  │ 用户代表组  │
   │ (数据标准)   │  │ (隐私伦理)  │  │ (社区声音)  │
   └─────────────┘  └────────────┘  └─────────────┘
                           │
                  ┌────────┴────────┐
                  │  执行团队(日常运营) │
                  └─────────────────┘
```

**理事会的席位分配（建议）：**
- 数据提供方代表：2-3席
- 数据主体代表：2-3席（关键！必须有用户/社区代表）
- 技术专家：2席
- 政府代表：1-2席
- 资金方代表：1席
- 独立人士：1-2席

## 第二章：技术建设阶段（第4-9个月）

### 2.1 平台架构设计

**最小可行产品（MVP）应包含的功能：**

1. **数据接入模块**——接收、验证、清洗来自不同来源的数据
2. **匿名化处理模块**——去除个人可识别信息（PII），应用差分隐私
3. **数据存储模块**——安全加密存储
4. **访问控制模块**——分级授权的数据访问
5. **审计追踪模块**——记录每一次数据访问

**不建议从头开发。** 参考现有开源方案（如DAMO、Dataverse）进行定制开发。

### 2.2 隐私保护技术方案

**最低标准：**
- 所有存储数据必须加密（AES-256）
- 数据传输必须加密（TLS 1.3）
- 数据访问必须日志化
- 应用差分隐私（ε ≤ 1.0）

**推荐方案：**
- 原始数据：加密存储在政府认可的云服务或本地服务器
- 分析数据：通过"数据沙箱"提供——研究者不能下载原始数据，只能在沙箱中运行分析
- 聚合数据：公开发布统计汇总

### 2.3 数据标准制定

**确保不同来源的数据可以互操作：**
- 采用国际标准（如FHIR for 医疗数据）
- 建立元数据标准（描述数据的数据，让使用者知道数据是什么）
- 数据质量验收流程

## 第三章：试点运营阶段（第6-18个月）

### 3.1 第一步：数据接入

**从哪里开始：**
- 从1-2个"低风险、高价值"的数据源开始
- 优先选择已经数字化且质量较好的数据
- 不要贪多——先跑通一个完整的数据流

### 3.2 第二步：用户接入

**第一批用户应该是：**
1. 学术研究机构（免费接入，换取研究成果和学术声誉）
2. 政府分析部门（作为公共服务义务）
3. 经严格筛选的AI公司（收费接入，验证商业模式）

**用户准入流程：**
```
提交申请 → 合规审查 → 理事会批准 → 签署协议 → 授予访问权限
```

### 3.3 第三步：价值创造——让数据产生实际效益

**公共价值（证明模式的正当性）：**
- 帮助公共卫生部门发现疾病模式
- 支持学术研究，发表论文
- 发布公共卫生报告，让公众受益

**经济价值（证明模式的可持续性）：**
- 向商业公司收取数据使用费
- 提供数据分析咨询服务
- 提供数据标注和验证服务

### 3.4 第四步：社区沟通

**不要只在技术圈子里打转。让社区参与进来：**
- 定期向数据主体汇报"你的数据带来了什么价值"
- 举办社区说明会，回答问题
- 建立投诉和反馈渠道

## 第四章：扩展阶段（第18-36个月）

### 4.1 数据源扩展

**从1-2个数据源到更多：**
- 扩展数据类型（从结构化数据到非结构化数据）
- 扩展数据来源（从医院到诊所、药店、实验室）
- 建立数据接入的"标准化对接流程"

### 4.2 用户群扩展

**从学术机构到商业机构到政府部门：**
- 制定分层定价策略
- 建立"数据信托认证"——获得认证的企业可以使用数据
- 确保扩展不损害数据主体的权利

### 4.3 地理扩展

**从一座城市到整个国家：**
- 将试点城市的成功经验文档化
- 与其他城市分享经验
- 推动国家级立法或政策支持

### 4.4 国际联网

**从一国到多国：**
- 建立跨境数据流动的法律框架
- 与国际数据信托网络（如有）互联互通
- 向其他国家输出"数据信托建设指南"

## 第五章：风险管理手册

### 5.1 风险清单与应对措施

| 风险 | 概率 | 影响 | 提前预防 | 事后应对 |
|------|------|------|---------|---------|
| **数据泄露** | 中 | 致命 | 加密+审计+最小权限 | 通知+补救+追责 |
| **政府换届** | 中 | 高 | 法治化而非人治化 | 启动外交保护 |
| **资金断裂** | 中 | 高 | 多元化资金来源 | 缩减规模+紧急募捐 |
| **技术故障** | 低-中 | 中 | 冗余+备份 | 恢复+复盘 |
| **法律诉讼** | 中 | 中 | 合规审查+保险 | 法律团队应对 |
| **社会反对** | 低 | 中 | 透明沟通 | 对话+调整 |
| **内部腐败** | 低 | 高 | 审计+透明 | 严惩+制度修补 |

### 5.2 最重要的三条铁律

1. **数据安全是生命线。** 一次泄露可能毁掉多年的信任建设。宁可功能少，不能不安全。
2. **透明度是最好的保护。** 所有决策可追溯、可审计、可质疑。没有不可见的数据使用。
3. **永远记住数据属于人民。** 数据托管机构是管理者，不是所有者。这个定位必须清晰。

## 第六章：失败预案

### 6.1 如果项目失败

任何革命都有牺牲，任何实验都有失败的可能。但数据信托的失败不能意味着数据主体的利益受损。

**有序退出机制：**
1. 提前在章程中规定解散条件和程序
2. 解散时确保所有数据的安全处理和归还
3. 将经验教训写成案例，供后来者参考

### 6.2 如何定义"成功"和"失败"

**成功的标准（从低到高）：**
| 级别 | 标准 | 时间预期 |
|------|------|---------|
| L1 | 数据信托存活超过3年 | 3年 |
| L2 | 持续产出公共价值（研究成果、公共报告） | 3-5年 |
| L3 | 实现财务自给自足 | 3-5年 |
| L4 | 被复制到其他地区/行业 | 5-7年 |
| L5 | 推动国家层面的数据公有制立法 | 7-10年 |

**失败不等于无用。** 即使项目最终关闭，只要过程是透明的、数据是安全的、经验是记录的，它就为后来的运动积累了重要的经验。
